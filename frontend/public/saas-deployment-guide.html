Part 1

Selling to Multiple Clients — Multi-Tenant SaaS

1.1 Understanding the Current Architecture

The ERP is already built as a multi-tenant application. Every record in every table carries a tenantId column. One running instance of the app can serve hundreds of companies completely isolated from each other — you do not need to deploy a separate server per client.

┌─────────────────────────────────────────────────────┐ │ Single Server / VPS / Cloud VM │ └─────────────────────────────────────────────────────┘ │ ┌────────┴────────┐ ┌─────────────┐ ┌──────────┐ │ Nginx (80/443) │ │ PostgreSQL │ │ Redis │ │ Reverse Proxy │ │ (shared) │ │ (shared) │ └────────┬────────┘ └─────────────┘ └──────────┘ │ │ ┌────────┴──────────┐ │ │ NestJS Backend │────────┘ Every query is scoped: │ (multi-tenant) │ WHERE "tenantId" = $1 └───────────────────┘ │ ┌────────┴──────────┐ │ Next.js Frontend │ ← Same UI for all clients └───────────────────┘ Client A (SAFIRA) → tenantId = "aaa-111" Client B (MONOPRIX) → tenantId = "bbb-222" Client C (PHARMACY) → tenantId = "ccc-333" All share the SAME database, backend, and frontend.
ℹ️ Key point: Because every table already has tenantId and the backend guards enforce it on every request, the app is SaaS-ready today. No code changes are required to serve a second client.

1.2 Deployment Models — Which One to Choose

ModelDescriptionBest ForCostIsolation
Shared SaaS
Recommended		 One server, one database, all clients share infrastructure. Queries scoped by tenantId. SME clients, <500 users total Lowest Logical (row-level)
Schema-per-tenant One PostgreSQL instance, each client gets their own schema (e.g. safira.sales). Mid-market clients needing stronger DB isolation Medium Schema-level
DB-per-tenant Each client has a completely separate PostgreSQL database. Enterprise / regulated industries (banking, health) High Full DB
Instance-per-tenant Full Docker stack per client on a dedicated VM. Very large clients or custom SLAs Very High Complete

For starting out, Shared SaaS is the right choice. You can migrate high-value clients to their own database later without changing the application code.

1.3 Client Onboarding Procedure

There are two ways to create a new client (tenant):

Option A — Self-Registration (recommended for public SaaS)

Clients register themselves at https://yourdomain.com/register. The app creates the tenant automatically.

Client fills in the registration form

Company name, subdomain slug, admin email, password, currency.

Backend creates the tenant record

The POST /api/v1/auth/register endpoint creates the tenant row, seeds default roles (Super Admin, Admin, Caissier, etc.), and returns access tokens.

Client lands on the dashboard

They configure their company settings, branches, warehouses, users, and products. The demo seed is NOT applied — they start fresh.

Send a welcome email

Add an email notification (via SendGrid / Mailgun) at the point of registration. The backend already has the user's email at this step.

Option B — Admin Provisioning (for onboarded enterprise clients)

You create the tenant manually from a superadmin CLI or admin panel:

# SSH into your server
ssh ubuntu@your-server-ip

# Run the registration API with curl
curl -X POST https://yourdomain.com/api/v1/auth/register \
  -H "Content-Type: application/json" \
  -d '{"companyName":"Client ABC","companySlug":"client-abc",
      "firstName":"Admin","lastName":"User",
      "email":"admin@clientabc.cm","password":"Temp@2024!",
      "currency":"XAF"}'
⚠️ Security: Always force a password change on first login for provisioned accounts. Add a mustChangePassword flag to the User model and redirect to a change-password page when set.

Checklist: Before Handing Over to Client

  • Tenant record created with correct currency and company name
  • Admin user created — credentials sent securely (not by plain email)
  • Branches and warehouses configured (or guide client to do so)
  • Tax codes set up for client's country
  • Demo data removed (or left if client requested a trial with sample data)
  • Custom logo/branding applied if white-label package sold
  • Subscription/billing record created in your billing system
  • Client added to your monitoring dashboard

1.4 White-Label Customisation

To brand the ERP for each client (their logo, colours, company name in the header):

🎨 Logo & Name

Store logoUrl and name in the Tenant table. The frontend fetches these from GET /settings/company and displays them in the sidebar and browser tab title.

🌐 Custom Domain

Point the client's domain (e.g. erp.clientabc.cm) to your server via DNS CNAME. Add that hostname to your Nginx config and issue an SSL certificate with Let's Encrypt.

🎨 Brand Colours

Add a primaryColor field to the Tenant model. Inject it as a CSS variable via a /api/v1/theme.css endpoint loaded by the frontend.

📧 Email From Address

Store SMTP credentials per tenant so invoice emails and notifications come from noreply@clientabc.cm instead of your own domain.

Adding a Custom Domain per Client (Nginx)

# /etc/nginx/sites-available/client-abc.conf
server {
    listen 443 ssl;
    server_name erp.clientabc.cm;

    ssl_certificate     /etc/letsencrypt/live/erp.clientabc.cm/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/erp.clientabc.cm/privkey.pem;

    location / {
        proxy_pass http://localhost:3000;  # Next.js frontend
        proxy_set_header Host $host;
    }
    location /api/ {
        proxy_pass http://localhost:4000;  # NestJS backend
    }
}

# Issue certificate
certbot --nginx -d erp.clientabc.cm

1.6 Data Isolation & Security

🔒 Row-Level Isolation

Every Prisma query includes { where: { tenantId } }. The backend middleware injects the tenantId from the JWT — clients can never see another client's data even if they know the ID.

🗝️ JWT Tenant Binding

The JWT payload contains { sub, tenantId, email, roles }. No query can be executed without a valid token that carries the correct tenantId.

🧱 PostgreSQL Row Security

For extra defence, enable PostgreSQL Row Level Security (RLS) on critical tables so the DB engine itself rejects cross-tenant reads even if application code has a bug.

📊 Audit Trail

The AuditInterceptor logs every mutating request (POST/PATCH/DELETE) with the userId and tenantId. This provides a complete record per client.

1.8 Client Offboarding

Export all client data

Run pg_dump with a WHERE tenantId = 'xxx' filter (using COPY TO per table or a custom export script) and deliver it to the client as CSV/Excel.

Disable the tenant account

Set tenant.isActive = false in the database. The login endpoint should check this flag and return a clear message.

Delete data after retention period

After 90 days (per your contract), hard-delete the tenant row — cascading foreign keys will remove all associated data. Document this policy in your service agreement.

Backup & Disaster Recovery

2.1 What Needs to Be Backed Up

ComponentData at RiskPriorityMethod
PostgreSQL DB All business data — sales, inventory, users, financials Critical pg_dump + WAL archiving
Uploaded files Product images, attachments, PDF invoices High rsync / S3 sync
.env file All secrets — JWT keys, DB credentials, API keys Critical Encrypted offline storage
Docker Compose + config Infrastructure definition (already in Git) Medium Git repository (GitHub)
Redis Sessions, job queues (transient — reconstructed on restart) Low Not required — Redis is ephemeral cache
Nginx config SSL certificates, virtual hosts High rsync + certbot renewal
🚨 Never skip: If the PostgreSQL database is lost without a backup, ALL client business data is gone permanently. Set up automated backups before going live with any paying client.

2.2 PostgreSQL Backup Procedures

Method 1 — pg_dump (Full logical backup, simplest)

Creates a SQL file that can restore the entire database on any PostgreSQL instance.

# Run from the server (not inside the container)
docker exec erp_postgres pg_dump \
  -U erp \
  -d erp \
  --format=custom \          # compressed binary format (faster restore)
  --no-owner \
  --no-acl \
  > /backups/erp_$(date +%Y%m%d_%H%M%S).dump

# Verify the backup file was created
ls -lh /backups/erp_*.dump

Method 2 — pg_basebackup (Full physical backup, for PITR)

Captures a binary snapshot of the entire PostgreSQL cluster. Combine with WAL archiving for Point-In-Time Recovery (restore to any second in time).

docker exec erp_postgres pg_basebackup \
  -U erp \
  -D /backups/basebackup_$(date +%Y%m%d) \
  --format=tar \
  --gzip \
  --checkpoint=fast \
  --progress

Backup Retention Policy

FrequencyRetentionStorage
Hourly incremental (WAL)48 hoursLocal + S3
Daily full (pg_dump)30 daysLocal + S3
Weekly full3 monthsS3 / Glacier
Monthly full1 yearS3 Glacier (cold)

2.3 Automated Backup Script

Save this as /opt/erp-backup/backup.sh on your server:

#!/bin/bash
# ── ERP Automated Backup Script ────────────────────────────────────────────
# Runs daily via cron. Dumps PostgreSQL, compresses, uploads to S3/offsite.
# ───────────────────────────────────────────────────────────────────────────

set -euo pipefail

# ── Config ──────────────────────────────────────────────────────────────────
BACKUP_DIR="/opt/erp-backup/dumps"
DB_CONTAINER="erp_postgres"
DB_NAME="erp"
DB_USER="erp"
RETAIN_DAYS=30
S3_BUCKET="s3://your-bucket/erp-backups"   # change this
TIMESTAMP=$(date +%Y%m%d_%H%M%S)
FILENAME="erp_${TIMESTAMP}.dump"
LOG_FILE="/opt/erp-backup/backup.log"

# ── Helpers ──────────────────────────────────────────────────────────────────
log() { echo "[$(date '+%Y-%m-%d %H:%M:%S')] $1" | tee -a "$LOG_FILE"; }

log "Starting backup: $FILENAME"

# ── 1. Create dump directory ─────────────────────────────────────────────────
mkdir -p "$BACKUP_DIR"

# ── 2. Dump PostgreSQL ──────────────────────────────────────────────────────
docker exec "$DB_CONTAINER" pg_dump \
    -U "$DB_USER" -d "$DB_NAME" \
    --format=custom --no-owner --no-acl \
  > "${BACKUP_DIR}/${FILENAME}"
log "DB dump complete: $(du -sh ${BACKUP_DIR}/${FILENAME} | cut -f1)"

# ── 3. Compress ──────────────────────────────────────────────────────────────
gzip "${BACKUP_DIR}/${FILENAME}"
COMPRESSED="${BACKUP_DIR}/${FILENAME}.gz"
log "Compressed: $(du -sh $COMPRESSED | cut -f1)"

# ── 4. Back up .env file (encrypted) ─────────────────────────────────────────
# Replace YOUR_GPG_KEY with your GPG key fingerprint
gpg --recipient YOUR_GPG_KEY --encrypt \
    /home/george-awamba/projects/ERP-PROJECT/.env \
    --output "${BACKUP_DIR}/env_${TIMESTAMP}.env.gpg"

# ── 5. Upload to S3 (requires aws-cli) ───────────────────────────────────────
if command -v aws &>/dev/null; then
    aws s3 cp "$COMPRESSED" "${S3_BUCKET}/${FILENAME}.gz"
    aws s3 cp "${BACKUP_DIR}/env_${TIMESTAMP}.env.gpg" "${S3_BUCKET}/env_${TIMESTAMP}.env.gpg"
    log "Uploaded to S3"
else
    log "WARNING: aws-cli not found — skipping S3 upload"
fi

# ── 6. Purge old local backups ───────────────────────────────────────────────
find "$BACKUP_DIR" -name "erp_*.dump.gz" -mtime +"$RETAIN_DAYS" -delete
log "Purged dumps older than $RETAIN_DAYS days"

log "✅ Backup complete: $FILENAME.gz"

Schedule with Cron

# Edit crontab
crontab -e

# Add these lines:
# Daily at 02:00 AM
0 2 * * *  /opt/erp-backup/backup.sh >> /opt/erp-backup/backup.log 2>&1

# Hourly WAL archiving (if using pg_basebackup)
0 * * * *  /opt/erp-backup/wal-archive.sh >> /opt/erp-backup/wal.log 2>&1

# Make script executable
chmod +x /opt/erp-backup/backup.sh
chmod 600 /opt/erp-backup/  # protect backup directory

2.4 Off-Site Storage

The 3-2-1 rule: 3 copies, on 2 different media, with 1 off-site. Options:

☁️ AWS S3

Industry standard. Use S3 Glacier for monthly archives (very cheap long-term storage). Enable versioning and MFA-delete on the bucket.

aws s3 cp backup.dump.gz \
  s3://your-bucket/backups/

🔧 Rclone (any cloud)

Open-source tool that syncs to 40+ cloud providers including Google Drive, Backblaze B2 (cheapest), OneDrive, Dropbox.

rclone copy /opt/erp-backup/dumps \
  backblaze:erp-backups/

🏠 Second VPS

Rsync backups to a cheap VPS in a different data centre. Simple and fully under your control.

rsync -avz /opt/erp-backup/ \
  backup-user@vps2:/backups/erp/

2.5 Restore Procedures — Step-by-Step

🚨 Before restoring: A restore overwrites all current data. Take a fresh backup of the current state first unless you're recovering a fully lost server.

Scenario A — Restore to the SAME server (accidental data deletion)

Stop the backend (to prevent writes during restore)

docker stop erp_backend

Drop and recreate the database

docker exec erp_postgres psql -U erp -c "DROP DATABASE erp;"
docker exec erp_postgres psql -U erp -c "CREATE DATABASE erp;"

Restore from the backup file

# Copy the backup file into the container
docker cp /opt/erp-backup/dumps/erp_20260611_020000.dump.gz erp_postgres:/tmp/

# Decompress
docker exec erp_postgres gunzip /tmp/erp_20260611_020000.dump.gz

# Restore
docker exec erp_postgres pg_restore \
  -U erp -d erp --clean --if-exists \
  /tmp/erp_20260611_020000.dump

Restart the backend

docker start erp_backend
docker logs erp_backend --tail 20

Verify

Log into the app and confirm data is restored. Check the most recent sale and inventory movement dates match expectations.

Scenario B — Restore to a FRESH server (server destroyed)

Provision a new VPS

Create a new Ubuntu 22.04 VPS with the same specs. Install Docker and Docker Compose:

curl -fsSL https://get.docker.com | sh
sudo apt-get install -y docker-compose-plugin

Clone the repository

git clone https://github.com/georgesbenz/ERP-PROJECT.git
cd ERP-PROJECT

Restore the .env file

# Download and decrypt the .env backup
gpg --decrypt env_20260611.env.gpg > .env

Start the infrastructure (DB only first)

docker compose up -d postgres redis
# Wait 15 seconds for Postgres to initialise
sleep 15

Download backup from S3 and restore

aws s3 cp s3://your-bucket/backups/erp_20260611_020000.dump.gz ./
gunzip erp_20260611_020000.dump.gz
docker cp erp_20260611_020000.dump erp_postgres:/tmp/
docker exec erp_postgres pg_restore \
  -U erp -d erp --clean --if-exists \
  /tmp/erp_20260611_020000.dump

Start all remaining services

docker compose up -d
# Monitor startup
docker compose logs -f --tail 30

Update DNS

Point your domain's A record to the new server IP. SSL certificate may need re-issuing (certbot --nginx -d yourdomain.com). Propagation takes 5–30 minutes.

2.6 RPO & RTO Targets

TermDefinitionTarget (daily backups)Target (hourly WAL)
RPO
Recovery Point Objective		 Maximum acceptable data loss. How old can the restored data be? Up to 24 hours of data loss < 1 hour of data loss
RTO
Recovery Time Objective		 Maximum acceptable downtime. How quickly must you be back online? 2 – 4 hours (manual restore) 30 – 60 min (scripted restore)
💡 Recommendation: For a SaaS product with paying clients, target RPO ≤ 1 hour and RTO ≤ 1 hour. This requires hourly WAL archiving and a pre-configured standby server or an automated restore script that runs with a single command.

2.7 Disaster Recovery Plan

Document this and keep it somewhere other than the server being recovered (e.g. printed copy, Google Doc, USB drive).

Incident TypeDetectionFirst ResponseRecovery Steps
Single record deleted by user User reports missing data Check audit log first (AuditInterceptor) Restore from last backup to a temp DB, extract the record, reinsert
Table truncated accidentally Immediate user reports Stop backend immediately Restore DB from same-day backup (Scenario A)
Server hardware failure Monitoring alert / users can't connect Check VPS provider dashboard Restore to new VPS (Scenario B), ~2 hrs
Database corruption NestJS errors, pg logs show corruption Stop all writes, take pg_dump of what's readable Restore from latest clean backup
Ransomware / hack Files encrypted or data exfiltrated Shut down server immediately, notify clients Provision fresh server from scratch, restore from off-site backup
Accidental docker compose down -v Immediate — data volumes deleted Do NOT restart Postgres yet Restore from last backup (Scenario A or B)

2.8 Testing Your Backups

A backup you have never tested is not a backup — it is hope. Run this procedure monthly:

Spin up a test environment

# On your laptop or a separate VPS
docker run -d --name pg-test \
  -e POSTGRES_USER=erp -e POSTGRES_PASSWORD=erp -e POSTGRES_DB=erp \
  -p 5433:5432 postgres:16

Restore the latest backup

docker cp erp_latest.dump pg-test:/tmp/
docker exec pg-test pg_restore -U erp -d erp /tmp/erp_latest.dump

Verify row counts match production

docker exec pg-test psql -U erp -d erp -c \
  "SELECT 'sales' AS t, COUNT(*) FROM sales
   UNION ALL SELECT 'inventory', COUNT(*) FROM inventory
   UNION ALL SELECT 'users', COUNT(*) FROM users;"

Compare the numbers against the production counts.

Log the test result

Record date, backup file used, row counts verified, and who performed the test. Keep this log off the production server.

Clean up

docker stop pg-test && docker rm pg-test

2.9 Monthly Operations Checklist

  • Verify last 7 daily backups exist in both local and off-site storage
  • Test-restore the most recent backup to a temporary database
  • Confirm row counts in test DB match production
  • Check disk usage on production server (alert if >75%)
  • Review Docker logs for errors: docker compose logs --since 7d | grep ERROR
  • Check SSL certificate expiry: certbot certificates
  • Rotate JWT secrets if any suspicious activity was detected
  • Review audit logs for unusual activity patterns
  • Update Docker images: docker compose pull && docker compose up -d
  • Review active client list — disable accounts for non-paying clients
  • Archive monthly backup to cold storage (S3 Glacier)
  • Review and update this checklist if procedures have changed
💡 Quick restore command (save this somewhere safe):
If you ever need to restore in an emergency, this single command restores the latest backup on a running server: 
LATEST=$(ls -t /opt/erp-backup/dumps/erp_*.dump.gz | head -1) && \
docker stop erp_backend && \
gunzip -k "$LATEST" && \
DUMP="${LATEST%.gz}" && \
docker cp "$DUMP" erp_postgres:/tmp/restore.dump && \
docker exec erp_postgres pg_restore -U erp -d erp --clean --if-exists /tmp/restore.dump && \
docker start erp_backend && \
echo "✅ Restore complete from $LATEST"
